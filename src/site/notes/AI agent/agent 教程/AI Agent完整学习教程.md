## 1. 学习路线与知识地图

AI Agent 不是一个单独的模型，而是以大语言模型为决策核心，组合上下文、工具、记忆、知识库、工作流和安全机制形成的应用系统。

建议按以下顺序学习：

1. **语言与工程基础**：Python、HTTP、JSON、异步编程、数据库、Docker、基本分布式系统知识。
2. **LLM 基础**：Transformer、Token、Embedding、上下文窗口、推理参数、微调和推理优化。
3. **LLM 应用**：Prompt、结构化输出、Function Calling、RAG。
4. **单 Agent**：ReAct、Plan-and-Execute、状态、工具、记忆、重试、终止条件。
5. **工作流框架**：先掌握直接调用模型 API，再学习 LangChain，重点掌握 LangGraph。
6. **多 Agent 与协议**：Supervisor、Peer-to-Peer、MCP、A2A、任务分配与结果聚合。
7. **生产工程**：评测、Trace、监控、安全、沙箱、部署、成本和发布策略。
8. **项目与系统设计**：至少完成一个带 RAG、工具、状态持久化、评测和部署的端到端项目。

判断是否需要 Agent，可以先问四个问题：

- 任务是否需要多步决策，而不只是一次模型生成？
- 执行路径是否会根据中间结果动态变化？
- 是否需要调用外部工具、读取实时数据或产生外部动作？
- 是否需要保存状态、支持中断恢复或人工审批？

如果答案大多为否，普通 Prompt、固定工作流或 RAG Chain 往往更简单、更稳定。不要为了使用 Agent 而使用 Agent。

## 2. 大语言模型基础

### 2.1 Transformer 与生成过程

Transformer 使用自注意力机制，让每个 Token 根据上下文计算与其他 Token 的关联。现代生成式大模型通常采用 Decoder-only 架构，按自回归方式逐 Token 生成文本：

1. 文本被 Tokenizer 切成 Token。
2. Token 被映射成向量表示，即 Embedding。
3. 多层 Transformer 计算上下文表示。
4. 输出层产生下一个 Token 的 logits。
5. logits 经 softmax 变成概率分布，再按解码策略选出下一个 Token。
6. 新 Token 加回上下文，重复上述过程。

大模型本质上预测“下一个 Token 最可能是什么”，并不是数据库，也没有内置的事实校验器。因此，流畅不等于真实，这也是幻觉的根本来源。

### 2.2 注意力与 KV Cache

自注意力将输入映射为 Query、Key、Value：

\[
Attention(Q,K,V)=softmax(\frac{QK^T}{\sqrt{d_k}})V
\]

生成第一个 Token 后，历史 Token 的 Key 和 Value 不会改变。KV Cache 会保存它们，后续生成时只计算新 Token 的 Query、Key、Value，避免重复计算全部历史。

KV Cache 的代价是显存占用随以下因素增长：

- 并发请求数；
- 上下文长度；
- 模型层数和隐藏维度；
- Key/Value 的数据精度。

常见优化包括量化 KV Cache、PagedAttention、前缀缓存、限制上下文、请求连续批处理和模型并行。

### 2.3 推理参数

- **temperature**：调整概率分布的尖锐程度。低温更确定，高温更多样。
- **top-p**：只从累计概率达到 `p` 的最小 Token 集合中采样。
- **top-k**：只保留概率最高的 `k` 个候选。
- **max_tokens**：限制最大输出长度，也是防止 Agent 长时间运行的基础约束。
- **frequency/presence penalty**：降低重复或鼓励新主题，具体语义依模型 API 而异。

推荐原则：

| 场景 | temperature 建议 | 目标 |
|---|---:|---|
| 工具选择、代码、SQL、分类 | 0～0.2 | 稳定和可复现 |
| RAG 问答、分析报告 | 0.1～0.4 | 准确优先，保留少量表达空间 |
| 创意写作、头脑风暴 | 0.7 以上 | 多样性 |

不要同时大幅调整 temperature 和 top-p。生产环境应通过评测集确定参数，而不是凭感觉设置。

### 2.4 训练与对齐

典型训练链路：

1. **预训练**：在海量语料上进行下一个 Token 预测，获得语言与知识能力。
2. **SFT**：使用指令—回答数据进行监督微调，让模型学会遵循任务格式。
3. **偏好对齐**：通过 RLHF、DPO 等方法，使输出更符合人类偏好和安全要求。

Function Calling 能力通常通过包含“何时调用工具、调用哪个工具、参数如何构造”的指令数据进行 SFT，并在偏好对齐阶段继续优化。

#### RLHF 与 DPO

RLHF 的典型流程是：收集多份候选回答并由人类排序 → 训练奖励模型 → 使用 PPO 等强化学习算法优化策略模型。优点是可以显式建模奖励并迭代采样，缺点是系统复杂、训练不稳定、算力和工程成本高。

DPO 直接使用“偏好回答/拒绝回答”数据优化策略模型，不单独训练奖励模型，也不需要在线强化学习。它更简单稳定，但效果高度依赖偏好数据覆盖度，对超出偏好数据分布的行为控制较弱。

#### SFT 数据质量

SFT 的上限主要由数据决定。数据构造应检查：

- 指令是否清晰、回答是否事实正确；
- 格式是否一致，是否覆盖真实生产输入；
- 任务类型、表达方式、语言和难度是否多样；
- 是否包含缺参、拒答、工具失败和安全边界样本；
- 是否去重，防止某类模板占比过高；
- 模型生成数据是否经过规则、模型和人工多级过滤。

Self-Instruct、回答重写和推理数据可以扩充数据，但不能替代真实用户数据。少量高质量数据通常优于大量未经校验的合成数据。

#### 灾难性遗忘

模型在垂直数据上微调后，可能改善领域任务却损害通用能力，这称为灾难性遗忘。原因是新任务梯度覆盖了原模型中对旧任务重要的参数方向。

缓解方法包括：

- 在微调数据中混入适量通用数据进行数据回放；
- 使用 LoRA 等参数高效微调，减少对基础权重的直接修改；
- 多任务训练，保持通用任务覆盖；
- 使用正则约束重要参数变化；
- 微调前后同时跑领域集与通用 Benchmark，不只看目标任务分数。

#### LoRA 与 QLoRA

LoRA 冻结原模型权重，只训练低秩增量矩阵：

\[
W'=W+BA
\]

其中秩 `r` 远小于原矩阵维度，因此显著减少训练参数和显存。QLoRA 进一步将基础模型量化后再训练 LoRA 适配器。

适合微调的场景：固定风格、垂直任务行为、特殊格式、领域语言。若需求只是补充可更新的事实知识，优先使用 RAG，而不是微调。

LoRA 的秩 `r` 决定增量矩阵的表达空间。`r` 太小可能欠拟合，太大增加显存、训练成本和过拟合风险。应通过验证集比较，而不是认为越大越好。QLoRA 的基础模型量化降低显存，但训练速度、数值稳定性和推理部署格式仍需单独验证。

### 2.5 推理优化概览

- **量化**：INT8、INT4、GPTQ、AWQ，以较小精度换取显存和吞吐收益。
- **FlashAttention**：减少注意力计算的显存访问开销。
- **PagedAttention**：以分页方式管理 KV Cache，提高显存利用率，vLLM 的核心能力之一。
- **连续批处理**：动态合并处于不同生成阶段的请求。
- **推测解码**：小模型先提出候选 Token，大模型批量验证。
- **蒸馏**：让小模型学习大模型输出分布或推理行为。
- **剪枝**：移除不重要权重或结构。

#### Prefill 与 Decode

推理可分为两个阶段：

- **Prefill**：一次处理全部输入 Token，计算密集，决定首 Token 延迟的一大部分。
- **Decode**：逐 Token 生成，依赖 KV Cache，更受显存带宽和缓存容量影响。

因此“长 Prompt + 短回答”和“短 Prompt + 长回答”的瓶颈不同。前者应优化上下文和前缀缓存，后者更关注 KV Cache、连续批处理和解码速度。

#### 量化方法的区别

- **INT8/INT4** 描述目标数值精度，不代表具体算法。
- **GPTQ** 使用近似二阶信息做逐层后训练权重量化，目标是降低重构误差。
- **AWQ** 根据激活分布识别更重要的权重通道，通过缩放保护少量显著权重。
- **Weight-only** 量化主要节省权重显存；若激活和 KV Cache 仍为高精度，长上下文并发下显存仍可能被缓存占满。

量化收益取决于硬件和 Kernel 支持。模型文件变小不等于实际吞吐一定提升，必须测首 Token 延迟、每 Token 延迟、吞吐、显存和任务精度。

#### 三种核心加速技术

- **FlashAttention**：通过分块计算和减少 HBM 读写优化注意力，核心收益来自 IO 感知，而不是近似注意力。
- **PagedAttention**：把 KV Cache 切成固定 Block 非连续存放，减少预分配浪费和显存碎片，并支持请求动态加入、退出批次。
- **Speculative Decoding**：草稿模型先生成多个候选 Token，目标模型一次验证；接受率越高，加速越明显，草稿模型与目标模型差异过大时收益会下降。

## 3. Prompt 与上下文工程

### 3.1 Prompt 的基本结构

一个可靠 Prompt 通常包含：

1. **角色与目标**：模型要完成什么任务。
2. **上下文**：完成任务所需的业务数据和背景。
3. **约束**：不能做什么、必须遵循什么规则。
4. **输出格式**：JSON Schema、字段解释、示例。
5. **工具规则**：何时调用、何时禁止调用、参数缺失时怎么办。
6. **不确定性处理**：信息不足时应询问、拒答还是检索。

示例：

```text
你是订单客服 Agent。
目标：回答订单状态问题，必要时调用 order_query。
约束：不得猜测订单状态；缺少订单号时先询问；退款操作必须人工确认。
输出：简洁中文，并标明信息来源。
```

约束应尽量转化为可检查规则。例如“回答不要太长”不如“最多 5 条，每条不超过 50 字”；“不要泄露隐私”还必须对应输出扫描和字段权限，而不能只写进 Prompt。

Few-shot 示例适合教模型理解边界和格式。示例应覆盖典型正例、易混淆反例和缺参场景，避免只给完全相同的模板导致模型机械模仿。示例中的事实数据应明确标为示例，防止被当成当前用户数据。

### 3.2 Prompt Engineering 与 Context Engineering

Prompt Engineering 关注如何写指令；Context Engineering 关注在有限上下文中，为当前步骤选择并组织最有价值的信息，包括：

- 系统规则；
- 当前用户请求；
- 对话摘要；
- 检索文档；
- 工具定义与工具结果；
- 用户长期记忆；
- 当前任务状态；
- 历史错误和约束。

生产 Agent 的主要难点通常不是一句 Prompt 怎么写，而是“当前步骤应该给模型什么、不给什么、以什么顺序给”。

常见上下文故障：

- **上下文污染**：旧任务信息残留到新任务。
- **指令冲突**：系统规则、用户请求、工具文档互相矛盾。
- **注意力稀释**：注入资料过多，关键证据被无关文本淹没。
- **中部遗忘**：重要信息位于超长上下文中间，模型利用率下降。
- **摘要漂移**：多次压缩后，限定条件逐步丢失或被改写。
- **工具膨胀**：一次暴露几十个相似工具，模型选择准确率下降。

应采用按需装载：先判断任务类型，再加载相关 Skill、工具和知识；工具执行完成后只保留必要结果，将大对象外置。每一类上下文都记录来源、版本和有效期。

### 3.3 结构化输出

自然语言输出不适合直接驱动程序。关键节点应使用 JSON Schema、Pydantic 或模型原生 Structured Output：

```json
{
  "intent": "query_order",
  "order_id": "A123",
  "need_human_approval": false,
  "confidence": 0.93
}
```

程序端仍须执行：

- JSON 解析；
- 类型和必填字段校验；
- 枚举、范围、长度校验；
- 业务权限校验；
- 失败后的修复或重试。

模型输出符合 JSON 格式，不代表它符合业务规则。

结构化输出失败可按以下顺序处理：原生 Schema 约束 → 本地解析和校验 → 将具体校验错误反馈模型修复 → 达到最大次数后降级或转人工。不要用正则从任意自然语言中“抢救”关键写操作参数。

### 3.4 Prompt 优化方法

优化应基于失败样本，而不是无限增加文字：

1. 收集失败 Trace。
2. 将失败归类为意图、检索、工具、格式、推理或安全问题。
3. 只修改对应层。
4. 在固定回归集上比较修改前后结果。
5. 检查质量、延迟、Token 和副作用。

Prompt 版本必须进入 Trace。修改一个系统 Prompt 可能改善平均正确率，却降低拒答或权限边界；所以 Prompt 变更应像代码一样经过评审、回归和灰度。

DSPy 一类工具可以将 Prompt 和模块参数化，基于训练样例与指标自动搜索更优组合，但仍依赖高质量数据和评价函数。

## 4. Agent 的定义与运行循环

### 4.1 Agent 的核心组成

一个完整 Agent 至少包含：

- **模型**：理解、推理、决策和生成。
- **指令与上下文**：目标、规则、当前状态。
- **工具**：访问外部世界或执行动作。
- **记忆**：保存会话状态、用户信息和经验。
- **规划器/路由器**：决定下一步。
- **执行器**：真正调用工具或节点。
- **观察与反馈**：读取结果、错误和环境变化。
- **终止机制**：判断完成、失败、超预算或转人工。

可抽象为循环：

```text
感知输入 → 更新状态 → 规划下一步 → 执行动作
       ↑                         ↓
       └──── 观察结果并判断是否继续 ────┘
```

感知模块不只是读取用户文本，还要把图片、文件、网页、传感器事件和工具返回转换成统一事件。生产中应保存原始输入引用与解析结果，区分“来源内容”“解析器推断”“模型推断”，避免后续把 OCR 错误或摘要错误当成原始事实。

Observation 应结构化，例如：

```json
{
  "source": "order_api",
  "status": "failed",
  "error_code": "TIMEOUT",
  "retryable": true,
  "attempt": 2,
  "data_ref": null
}
```

结构化 Observation 能让规划器区分空结果、业务拒绝和系统故障，也便于规则节点先处理确定性错误。

### 4.2 Agent 与固定工作流的区别

| 类型 | 路径 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| 单次 LLM | 一次调用 | 简单、便宜 | 能力有限 | 摘要、分类、改写 |
| Chain | 固定步骤 | 稳定、易测 | 不适应动态情况 | 固定数据处理 |
| Workflow | 显式分支/并行 | 可控、可恢复 | 设计成本较高 | 企业流程 |
| Agent | 模型动态决策 | 灵活 | 不确定、成本高 | 开放式多步任务 |

生产实践通常不是“完全自主 Agent”，而是“确定性工作流包围少量模型决策节点”。高风险步骤越多，流程越应显式化。

### 4.3 自主性等级

可以把自主性分为：

1. 只生成建议，不执行。
2. 可调用只读工具。
3. 可执行低风险、可撤销动作。
4. 高风险动作执行前需要人工审批。
5. 在明确权限和预算内自主运行。

权限应按最小授权设计，而不是把所有工具交给模型再靠 Prompt 约束。

Human-in-the-Loop 不应只在最后加一个“确认按钮”，而应依据风险触发。风险可由副作用大小、可撤销性、金额、数据敏感度、模型置信度、工具异常和策略命中共同评分。高风险动作应展示动作摘要、关键参数、证据和预计影响，让人类知道自己在批准什么。

## 5. 规划、推理与流程控制

### 5.1 ReAct

ReAct 将推理与行动交替进行：

```text
分析当前状态 → 选择动作 → 调用工具 → 观察结果 → 再决策
```

优点：适合路径未知、需要边查边做的任务。缺点：调用次数难预测，可能重复调用或陷入循环。

适合 ReAct 的任务具有“下一步依赖刚获得的信息”这一特征，例如搜索研究、故障排查和交互式数据探索。若步骤在开始前已经明确，固定工作流通常更稳定。

工程中不应依赖模型暴露完整思维链，而应记录可审计的简短决策摘要、工具选择原因和结构化状态。

### 5.2 Plan-and-Execute

先生成整体计划，再由执行器逐步完成，必要时重新规划。

适合：步骤多、依赖明确、任务耗时较长的研究或数据处理。相较 ReAct，计划更清晰，也更容易做并行调度；缺点是初始计划可能基于错误假设。

可靠实现应包含：

- 计划步骤的结构化表示；
- 步骤依赖关系；
- 每步状态与产物；
- 执行失败后的局部重试；
- 事实变化后的重新规划；
- 最大重规划次数。

计划不是自然语言待办列表，而应是可执行的数据结构。每步至少包含 ID、目标、依赖、预期产物、执行者、状态和验证条件。重新规划时应保留已验证结果，避免从头重复全部工作。

### 5.3 CoT、Self-Consistency 与任务分解

- **Chain-of-Thought**：将复杂问题拆为中间步骤，适合数学和逻辑推理。
- **Self-Consistency**：以不同采样路径生成多个候选，再投票或聚合。
- **任务分解**：将大任务拆为可验证的小任务，降低单次推理难度。

这些方法能提升某些任务的准确率，但会增加 Token 和延迟。生产系统更应强调中间结果可验证，而不是单纯生成更长的推理文本。

Self-Consistency 只有在候选路径具有足够多样性、且最终答案可可靠聚合时才有意义。可用自适应策略控制成本：先生成少量候选；如果答案一致则停止，分歧大时再增加采样；简单样本用小模型，最终仲裁再用强模型。

### 5.4 Reflexion 与反思

Reflexion 在失败后生成一段可复用的反思，例如“错误原因、下次策略、不可重复的动作”，再加入后续上下文。

有效反思应：

- 绑定具体任务和证据；
- 简短、结构化；
- 有有效期或置信度；
- 避免把一次偶然失败固化成永久规则；
- 通过评测确认确实改善后续表现。

Reflexion 的典型闭环是 `执行 → 评价 → 生成反思 → 加入 Reflection Memory → 再执行`。评价器必须基于测试、规则或外部反馈，而不能只问同一个模型“你做得对不对”，否则容易形成自我确认。

### 5.5 Tree of Thoughts

ToT 同时探索多个候选思路，对节点评分并剪枝，适合搜索空间明确且中间状态可评价的任务。其代价是模型调用量快速增长，不适合普通问答。

若分支数为 `b`、深度为 `d`，最坏节点数接近指数增长。工程上通过限制 `b` 和 `d`、每层只保留 Top-K、低分早停、缓存相同状态，以及用小模型评分来控制成本。

### 5.6 有限状态机与图工作流

高可靠流程可显式建模为状态机：

```text
INIT → UNDERSTAND → RETRIEVE → DECIDE → EXECUTE → VERIFY → DONE
                           ↘ NEED_APPROVAL ↗
                           ↘ FAILED
```

设计原则：

- 每个状态职责单一；
- 状态转移条件可测试；
- 非法转移默认拒绝；
- 状态持久化，支持中断恢复；
- 外部副作用动作使用幂等键；
- 为循环设置步数、时间和费用上限。

### 5.7 终止条件

至少包含：

- 模型明确返回完成；
- 到达终态；
- 最大步骤数；
- 最大连续失败次数；
- 超时；
- Token 或费用超预算；
- 相同动作重复；
- 需要人工输入；
- 安全策略触发。

## 6. 工具调用与 Function Calling

### 6.1 工作原理

Function Calling 并不是模型直接执行函数。完整流程是：

1. 应用把工具名称、描述和 JSON Schema 发给模型。
2. 模型决定是否调用工具，并生成工具名与参数。
3. 应用校验参数、权限和业务规则。
4. 应用执行真实函数。
5. 工具结果作为 Tool Message 返回模型。
6. 模型基于结果生成回复或继续调用工具。

因此，模型负责“提出调用请求”，应用负责“是否允许以及如何执行”。

一次调用至少应有 `tool_call_id`，用于把模型请求、真实执行、工具结果和后续模型消息关联起来。多个工具并行时，结果必须按对应 ID 回填，不能依靠返回顺序猜测。

模型“不调用工具”也可能是正确决策。工具选择评测应同时包含：应该调用、可以不调用、禁止调用、缺参先询问四类样本。

### 6.2 工具 Schema 设计

一个好工具应具备：

- 名称清晰，如 `get_order_status`，避免 `process_data`。
- 描述明确说明用途、调用时机和禁用场景。
- 参数少且职责单一。
- 类型、枚举、必填项、长度和格式清晰。
- 对易混工具写出边界差异。
- 返回结构稳定，错误可机器解析。

```json
{
  "name": "get_order_status",
  "description": "根据订单号查询订单状态。仅用于查询，不会修改订单。",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "完整订单号，例如 A20260712001"
      }
    },
    "required": ["order_id"],
    "additionalProperties": false
  }
}
```

### 6.3 工具粒度

- 过粗：参数多、权限大、失败原因不透明，模型难以正确调用。
- 过细：调用链过长、延迟和成本增加、状态一致性变复杂。

对模型暴露的工具应贴近一个清晰业务意图。底层可以由多个原子服务组成，但可封装为经过权限审计的领域工具。

判断粒度时看四个维度：是否可独立授权、是否可独立重试、是否有清晰输入输出、失败后能否判断副作用。把“查询订单并退款并发短信”做成一个工具，会导致权限过大且中间失败难以补偿；拆成几十个字段级工具，又会让模型规划复杂度和调用成本失控。

### 6.4 参数校验与错误处理

校验必须发生在应用端，至少分三层：

1. **Schema 校验**：类型、必填、枚举、格式。
2. **业务校验**：订单是否属于当前用户、金额是否越界。
3. **安全校验**：权限、注入、路径、SQL、敏感数据。

错误返回应结构化：

```json
{
  "error_code": "INVALID_ORDER_ID",
  "message": "订单号必须为 14 位字母数字组合",
  "retryable": true,
  "suggestion": "请向用户确认完整订单号"
}
```

重试策略：

- 参数错误：把明确错误返回模型，允许有限次数修正。
- 超时或限流：指数退避加随机抖动。
- 鉴权失败：不要盲目重试，转登录或人工处理。
- 服务不可用：降级到缓存、备用服务或告知用户。
- 业务拒绝：作为最终结果，不应重试。

重试前必须判断上一次调用是否可能已经成功。例如支付 API 客户端超时，不代表服务端没有扣款。此时应先用幂等键查询原操作状态，而不是直接再次支付。

为避免模型在同一错误上循环，可记录错误指纹 `(tool, normalized_args, error_code)`；相同指纹连续出现达到阈值后，禁止继续重试并切换方案或转人工。

### 6.5 并行与串行调用

无依赖、只读操作可并行，如同时查询天气和航班。有依赖或写操作应串行，如先创建订单再支付。

并发安全需要：

- 幂等键；
- 乐观锁或版本号；
- 唯一约束；
- 事务或 Saga 补偿；
- 重复请求检测；
- 明确的依赖 DAG。

### 6.6 异步长任务

不要让模型调用长时间阻塞的工具。应改为：

1. `submit_job` 返回 `job_id`。
2. 后台队列执行任务。
3. Agent 轮询 `get_job_status`，或由 Webhook/事件通知。
4. 完成后读取产物。

任务必须支持超时、取消、进度、重试、幂等和结果过期。

异步任务状态可统一为 `queued → running → succeeded/failed/cancelled/expired`。状态迁移应由任务系统维护，模型只能查询或提出取消请求，不能自行伪造完成状态。

### 6.7 大型工具结果

工具不应把数万行数据直接塞回上下文。可采用：

- 服务端分页、过滤、聚合；
- 只返回字段白名单；
- 先保存为文件或对象存储，再返回引用；
- 对结果分层摘要；
- 将结果写入检索索引，按需读取；
- 为每个工具设置 Token 预算。

摘要不能替代原始证据。关键决策应能追溯到原记录。

### 6.8 代码执行与 Text-to-SQL

代码执行必须在沙箱中进行：非特权用户、只读基础镜像、临时文件系统、CPU/内存/时间限制、网络默认关闭、依赖白名单、密钥隔离和审计日志。容器只是隔离手段之一，不等于完整安全边界。

Text-to-SQL 推荐链路：

```text
用户问题 → 选择相关 Schema → 生成 SQL → AST/规则校验
→ 强制只读与 LIMIT → EXPLAIN/成本检查 → 受限账号执行
→ 结果校验 → 生成回答
```

禁止直接信任模型 SQL；生产库使用只读副本、表列白名单、行级权限、查询超时和扫描量限制。

## 7. MCP、A2A 与 Skill

### 7.1 MCP

MCP（Model Context Protocol）用于标准化 AI 应用与外部能力之间的连接。典型角色包括：

- **Host**：承载用户交互和 Agent 的应用。
- **Client**：Host 内连接某个 MCP Server 的协议客户端。
- **Server**：暴露 Tools、Resources、Prompts 等能力。

MCP 解决的是“如何发现、描述和调用外部上下文/工具”的连接标准问题。Function Calling 是模型 API 层的工具选择机制，两者可以配合：MCP Client 把 Server 工具转换为模型可见的工具定义，模型选中后再由 Client 通过 MCP 调用。

MCP 的三类核心能力：

- **Tools**：可执行动作，由模型或应用选择调用。
- **Resources**：可读取的文件、数据库记录、文档或其他上下文。
- **Prompts**：Server 提供的可复用提示模板或工作流入口。

典型会话包含初始化与能力协商、列出能力、读取资源/调用工具、返回结果和关闭连接。Client 不应假设所有 Server 支持相同能力，应根据协商结果工作。

工具描述仍要遵循 Function Calling 的设计原则：单一职责、明确副作用、严格 Schema、结构化错误。协议标准化了连接方式，但不会自动保证工具安全或质量。

本地 Server 适合访问本机文件和开发工具；远端 Server 适合共享企业服务。远端部署必须考虑认证、授权、租户隔离、传输安全、超时、限流和审计。

本地进程常通过标准输入输出传输，部署简单但生命周期依赖 Host；远端服务可通过 HTTP 和流式传输支持多人共享与长任务，但必须避免把长期密钥直接交给模型。认证应发生在 Host/Client 和 Server 之间，并使用短期、最小作用域凭据。

### 7.2 A2A

A2A（Agent-to-Agent）关注不同 Agent 之间的互操作：

- Agent 通过能力卡片描述身份、能力、接口和安全要求；
- 调用方提交任务；
- 被调用方返回任务状态、进度、消息和产物；
- 支持长任务、异步更新和流式结果；
- 调用方不需要了解对方内部模型、工具或框架。

A2A 的关键不是让两个模型互相聊天，而是围绕一个可追踪任务协作。任务应具有唯一 ID、明确输入、当前状态、消息历史和产物引用。常见生命周期可抽象为：

```text
submitted → working → input-required → working
          ↘ completed / failed / cancelled
```

当被调用 Agent 缺少信息时，应进入 `input-required`，而不是编造参数。长任务通过状态查询、事件或流式更新报告进度；最终大文件使用 Artifact 引用传递，不把全部内容塞进消息。

Agent Card/能力卡片需要描述能力、输入输出形式、访问地址、认证方式和支持的交互模式。发现能力不等于获得权限，真正执行仍需身份认证与授权。

简化对比：

| 技术 | 主要连接对象 | 核心问题 |
|---|---|---|
| Function Calling | 模型 ↔ 应用函数 | 模型如何提出工具调用 |
| MCP | AI 应用 ↔ 工具/资源提供方 | 外部能力如何标准接入 |
| A2A | Agent ↔ Agent | 异构 Agent 如何协作 |

### 7.3 Skill

Skill 是可复用的任务能力包，通常不只是一条 Prompt，还可以包含：

- 适用场景与触发规则；
- 操作步骤；
- 工具使用规范；
- 模板、脚本和参考资料；
- 输入输出契约；
- 示例和失败处理；
- 版本信息与权限要求。

Tool 是原子动作，Skill 是完成一类任务的方法。MCP 负责连接能力，Skill 负责组织知识和工作流。

高质量 Skill 应满足：

- **可发现**：名称、描述和触发条件能与任务准确匹配；
- **可执行**：步骤明确，依赖、工具和输入齐全；
- **可组合**：输入输出契约稳定，不把内部状态隐式泄露给其他 Skill；
- **可验证**：有测试样例、成功标准和失败处理；
- **可治理**：声明版本、权限、维护者、兼容性和废弃时间；
- **渐进加载**：先加载摘要，命中后再加载完整说明和参考资料。

Skill 质量可从触发准确率、任务成功率、平均步骤数、工具错误率、Token/费用和跨版本回归率评估。一个 Skill 即使最终成功，如果经常误触发或需要大量无效步骤，也不是高质量 Skill。

Skill 库应支持注册、检索、版本管理、依赖声明、权限审核、质量评测和废弃策略。避免一次向模型注入全部 Skill 正文，应先根据任务检索候选 Skill，再按需加载详细说明。

版本升级应区分兼容变更和破坏性变更。生产任务最好记录实际使用的 Skill 版本，使失败 Trace 可复现；新版本先运行回归集和灰度流量，再替换默认版本。

## 8. 记忆系统

### 8.1 四层记忆架构

可按用途划分：

1. **即时上下文**：当前输入和当前步骤工具结果。
2. **工作记忆**：当前会话的消息、计划、任务状态和临时变量。
3. **长期语义记忆**：用户偏好、事实、历史事件，以 KV、关系库或向量库保存。
4. **程序性/反思记忆**：完成任务的方法、失败经验和高阶规律。

不是所有对话都应该写入长期记忆。写入前应判断价值、置信度、敏感性、作用域和有效期。

还可以按认知用途细分：

- **语义记忆**：稳定事实，如用户偏好、组织规则、实体属性。
- **情节记忆**：带时间和场景的事件，如“上周用户取消了某订单”。
- **程序性记忆**：完成任务的方法和操作步骤。
- **反思记忆**：从多次事件中提炼出的高阶规律。

稳定偏好不应每次从数千条情节记录里碰运气检索。写入阶段应把高价值信息从情节记忆“提升”为结构化语义记忆；原始事件仍保留为可追溯证据。

### 8.2 上下文窗口管理

常见策略：

- 滑动窗口：只保留最近消息，简单但会丢失早期约束。
- 分层摘要：将旧消息压缩成会话摘要。
- 重要度保留：始终保留目标、约束、关键决策和未完成事项。
- 检索式记忆：按当前问题召回相关历史。
- 状态外置：计划和业务状态存数据库，不依赖聊天记录还原。

记忆爆炸的本质是信息密度与 Token 预算的冲突。可使用多阶段压缩：

1. 计算当前 Token，并预留模型输出空间。
2. 对消息做重要性评分。
3. 删除重复、低价值、过期信息。
4. 对较早历史生成结构化摘要。
5. 保留最近若干轮原文以及目标、约束、决策、未完成事项。
6. 超过预算时再做更激进压缩或按需检索。

重要性可综合以下信号：

\[
importance = w_c content + w_n novelty + w_a access + w_r recency + w_u userSignal
\]

其中 `userSignal` 表示用户明确说“记住”“以后都这样”等强信号。权重和阈值必须通过真实会话评测，不能直接照搬固定比例。

建议将上下文分为独立预算区：系统与安全规则、任务状态、近期对话、检索知识、长期记忆、工具结果和预期输出。任何一个区都不应无限挤占其他区。

推荐上下文顺序：

```text
系统规则
当前任务目标与约束
结构化任务状态
相关长期记忆
相关知识检索结果
最近若干轮对话
当前用户输入
```

对于 200 轮长对话，“用户之前的核心需求”应使用三层保障：写入时抽取并 Pin；压缩时明确要求保留需求和约束；若仍缺失，再从情节记忆中按关键词与语义混合检索。

### 8.3 长期记忆存储

- **关系库/KV**：存精确事实、用户配置、任务状态。
- **向量库**：按语义检索非结构化事件和文本。
- **对象存储**：存原始长文档、附件和产物。
- **图数据库**：存实体关系和复杂关联。

常用检索评分可综合：

\[
score = w_r \cdot relevance + w_i \cdot importance + w_t \cdot recency
\]

召回后还应去重、重排、过滤权限，并限制注入上下文的数量。

#### MemGPT/Letta 的虚拟上下文思想

这类架构借鉴操作系统虚拟内存，把有限的模型上下文当作主存，把外部记忆库当作磁盘：

- Main Context 保存系统规则、工具、当前对话和核心记忆；
- Recall Storage 保存完整历史事件；
- Archival Storage 保存长文档和可语义检索知识；
- Agent 通过 memory search/insert/replace 等工具主动换入、写出或修改记忆。

它与传统 RAG 的区别不仅是主动与被动：传统 RAG 通常在每次请求前固定检索且主要只读；虚拟上下文允许 Agent 在任务过程中决定检索时机、连续多次检索并写入新记忆。代价是更多模型调用、更高延迟，以及错误换入/换出的风险，所以核心记忆仍需规则保护。

面对数千条历史消息，应做冷热分离：稳定画像作为热数据直接加载，高价值事件保留在检索索引，低价值原文进入低成本归档；先通过反思提炼信息，再按需召回原始证据。

### 8.4 反思记忆

系统可周期性从大量事件中提炼稳定规律，例如用户长期偏好。但要防止错误归纳：

- 保留来源事件；
- 记录生成时间和置信度；
- 新证据可覆盖旧结论；
- 重大结论需要用户确认；
- 低价值反思定期淘汰。

Reflection 可按“事件积累 → 重要度达到阈值 → 生成候选规律 → 绑定来源 → 验证/确认 → 写入”的流程执行。不要每轮对话都反思，否则不仅昂贵，还会产生大量互相矛盾的伪规律。

例如，用户一次购买低糖食品不能直接推断为长期饮食偏好；只有多次行为或用户明确确认后，才应写成高置信度语义记忆。反思结论应允许撤销、覆盖和过期。

### 8.5 隔离、隐私与删除

记忆必须带作用域：`tenant_id`、`user_id`、`team_id`、`thread_id`。查询过滤应在存储层强制执行，不能只依靠 Prompt。

PII 保护包括：写入前检测、脱敏或加密、字段级访问控制、密钥分离、访问审计、保留期限和删除流程。

“删除我的数据”必须覆盖关系库、向量库、缓存、对象存储、索引和可删除备份，并保留不含原始内容的删除审计记录。

团队共享记忆应拆分作用域：个人私有、团队共享、组织公共。查询时先鉴权再检索，而不是先召回后让模型自行忽略越权内容。向量数据库也必须在检索条件中强制加入租户和作用域过滤。

## 9. RAG：让 Agent 使用外部知识

### 9.1 基本流程

RAG 分为离线索引和在线问答。

离线：

```text
采集文档 → 清洗解析 → 分块 → 元数据 → Embedding → 建索引
```

在线：

```text
理解问题 → 改写/路由 → 检索 → 重排 → 组装上下文
→ 生成答案 → 引用与事实校验
```

离线链路最容易被低估的是解析和版本管理。PDF 多栏、扫描件、表格、页眉页脚会破坏文本顺序；需要根据文档类型组合版面解析、表格抽取、OCR 或多模态模型。每个 Chunk 应保存 `document_id`、`chunk_id`、标题路径、页码、版本、权限、更新时间和内容哈希。

文档更新时应采用可追踪的增量流程：计算内容哈希 → 只重建变化块 → 删除失效向量 → 更新版本别名。更换 Embedding 模型或维度后，旧向量不能与新向量混用，通常需要新建索引并完成全量重嵌入后切换。

### 9.2 文档分块

分块决定召回单位。常见策略：

- 固定 Token 长度加重叠；
- 按标题、段落、表格、代码块等结构切分；
- 按语义边界切分；
- 父子块：小块负责召回，大块负责提供上下文。

块过小会丢上下文，过大会降低匹配精度并浪费 Token。应依据文档结构、问题粒度和评测结果调整。

常见选型：

| 数据类型 | 推荐策略 | 原因 |
|---|---|---|
| FAQ、短问答 | 每个问答为一个块 | 自然检索单元明确 |
| 技术文档 | 标题层级 + 递归字符切分 | 保留章节语义和代码边界 |
| 法律/制度 | 条款级切分 + 父子块 | 既要精确命中又要保留上下文 |
| 代码仓库 | 函数、类、模块和符号关系 | 固定字符切分会破坏语法单元 |
| 日志 | 时间窗口或事件模板 | 语义段落不明显 |

Parent-Child 策略使用小块做索引、大块提供生成上下文。例如用 300 Token 子块召回，命中后返回对应 1200～1500 Token 父块，兼顾召回精度与语义完整性。

元数据建议包含来源、标题层级、更新时间、权限、文档类型、语言和版本。

### 9.3 检索优化

#### 混合检索

向量检索擅长语义相似，BM25 擅长关键词、编号和专有名词。混合检索融合两者，常比单一向量检索稳定。

两路得分尺度通常不同，可使用 Reciprocal Rank Fusion：

\[
RRF(d)=\sum_i \frac{1}{k+rank_i(d)}
\]

RRF 只依赖排名，不依赖各检索器分数归一化，因此实现简单且鲁棒。产品型号、错误码、API 名称和人名更依赖稀疏检索；同义表达和自然语言问题更依赖稠密检索。

#### Query 改写

- 消除代词，补全上下文；
- 将复杂问题拆为子问题；
- 生成多个检索表达；
- 使用假设文档扩展语义；
- 根据意图路由到不同知识库。

改写不能改变用户原意，最好同时保留原查询参与检索。

常见改写失败包括：添加用户未提供的条件、把多个子问题合并丢失细节、将缩写错误展开。应记录改写前后 Query，并在评测中单独计算改写带来的召回增益和语义漂移率。

#### Reranker

初次检索追求召回率，取较多候选；Cross-Encoder 或 LLM Reranker 再逐项评估问题与文档相关性，选择少量高质量块进入上下文。

Bi-Encoder 将 Query 和文档分别编码，文档向量可预计算，适合大规模 ANN 召回；Cross-Encoder 把 Query 与候选文档放入同一序列，使 Token 充分交互，精度更高但不能全库计算。因此常用“Top-20～100 粗召回 → Top-3～10 精排”的两阶段架构。

#### RAG 故障排查

当“文档明明有答案，系统却回答不知道”时，应按链路定位：

1. 文档是否成功解析并进入索引？
2. 正确内容是否被切到完整 Chunk 中？
3. 原 Query 和改写 Query 的 Top-K 是否召回目标块？
4. 目标块是否被 Reranker 错误过滤？
5. 上下文拼装时是否因 Token 超限被截断？
6. Prompt 是否要求模型只基于上下文，而上下文格式又使证据不易识别？
7. 模型是否正确引用并综合证据？

只有第 7 步属于生成问题。很多所谓“模型不知道”，根因其实在解析、分块或召回。

### 9.4 向量数据库选型

选型维度包括：

- 数据规模和查询并发；
- 过滤能力和多租户隔离；
- 索引类型、召回率和延迟；
- 混合检索支持；
- 更新、删除和一致性；
- 云托管或本地部署；
- 运维成本和现有技术栈。

小规模项目可用 PostgreSQL + pgvector；搜索系统可考虑 Elasticsearch/OpenSearch；大规模向量检索可使用 Milvus、Qdrant、Weaviate 等。没有脱离业务指标的“最佳向量数据库”。

向量索引也存在速度、内存与召回率权衡。HNSW 查询快、召回高但内存占用较大；IVF 先聚类再搜索部分倒排单元，规模大时更节省资源但需要调节 `nprobe`。索引参数必须在本业务数据上用 Recall@K 和延迟共同测试。

### 9.5 RAG 评测

分层评测：

- **检索**：Recall@K、Precision@K、MRR、nDCG。
- **生成**：答案正确性、相关性、完整性、引用准确性。
- **忠实度**：回答是否由检索上下文支持。
- **系统**：延迟、Token、成本、拒答率。

常用指标含义：

- `Recall@K`：所有相关文档中有多少被 Top-K 找到，衡量漏召回。
- `Precision@K`：Top-K 中有多少真正相关，衡量噪声。
- `MRR`：第一个相关结果出现得越靠前越好。
- `nDCG`：同时考虑多级相关性和排名位置。
- `Faithfulness`：回答中的断言是否能被上下文支持。
- `Answer Relevance`：回答是否直接回应问题。
- `Context Relevance`：注入上下文是否真正有用。
- `Citation Correctness`：引用是否存在且支持对应结论。

RAG 评测必须把检索和生成分开，否则最终答案错误时无法判断该改 Embedding、Chunk、Reranker、Prompt 还是模型。

评测集应包含正常问题、模糊问题、无答案问题、跨文档问题、时效问题、权限问题和对抗性问题。

### 9.6 Graph RAG、Self-RAG 与 Agentic RAG

- **Graph RAG**：将实体和关系建图，适合跨文档、多跳关系推理。
- **Self-RAG**：模型判断是否需要检索，并对证据和回答进行自检。
- **Agentic RAG**：Agent 动态选择知识库、搜索工具、查询策略和迭代次数。

高级 RAG 提升了灵活性，也增加延迟和不可预测性。应先把基础检索、元数据、评测和引用做好。

## 10. Agent 框架与技术选型

### 10.1 先理解框架解决什么问题

框架通常提供模型适配、Prompt、工具、状态、持久化、流式输出、工作流、回调和可观测性。框架不能替代业务建模、安全设计和评测。

### 10.2 直接调用 API

优点：依赖少、行为透明、容易优化；缺点：工具循环、状态、重试、Trace 等需要自行实现。

适合简单 Agent、性能敏感系统，以及希望完全掌控执行逻辑的团队。

### 10.3 LangChain

核心能力包括模型抽象、Prompt、Tool、Retriever、Output Parser 和 LCEL 组合。

LCEL 使用类似管道的方式组合 Runnable，支持串行、并行、分支、流式和回调。适合快速搭建 LLM Chain 和 RAG，但复杂循环、长期状态和审批流程更适合 LangGraph。

LangChain 的 Output Parser 负责把文本解析成结构化对象；模型原生 Structured Output/Function Calling 则在生成阶段约束结构。生产中优先使用模型原生结构化能力，再由 Pydantic 等做二次校验。

LCEL 的价值是统一组件接口和组合方式，但过度抽象可能增加调试难度。简单链路如果直接 API 调用更清楚，就不必强行包装成多层 Chain。

### 10.4 LangGraph

核心概念：

- **State**：图中共享的结构化状态。
- **Node**：读取状态并返回状态更新的计算单元。
- **Edge**：固定控制流。
- **Conditional Edge**：根据状态动态路由。
- **Checkpoint**：保存每一步状态，支持恢复和时间旅行调试。
- **Interrupt/Resume**：在指定节点暂停，等待人工输入后继续。
- **Subgraph**：封装可复用子流程。

合理的 State 只保存跨节点必要数据，不把所有原始日志都塞进去。并发节点更新同一字段时应定义明确的 reducer。

State 设计建议：

- 业务事实与展示文本分开；
- 可追加列表使用 reducer，单值字段明确覆盖规则；
- 保存工具结果引用，不保存超大原文；
- 错误记录包含节点、类型、次数和是否可重试；
- 计划、当前步骤、预算和审批状态显式建模。

Checkpoint 不只是聊天记忆。它保存图在某一步的完整状态，使进程重启后继续执行，也支持查看历史快照和从旧状态创建新的执行分支。开发环境可用内存存储，生产环境使用 PostgreSQL 等持久化后端，并设置 `thread_id` 与租户隔离。

Human-in-the-Loop 的常见模式：

- 在付款、发送、删除节点之前中断；
- 人工修改模型生成的参数后继续；
- 让人工在多个路由选项中选择；
- 审批拒绝后进入补充信息或取消分支。

Subgraph 用于封装 RAG、审核、报表等可复用流程。父图与子图应通过明确字段映射交换状态，私有字段留在子图内部，避免不同模块隐式耦合。

节点级容错应区分：可重试异常使用退避策略；业务错误走条件边；不可恢复异常进入失败终态；外部写操作恢复时先检查幂等状态。全图设置 recursion/step limit，节点设置超时，两者共同防止无限运行。

适用场景：多步骤、有循环、需持久化、需人工审批、需要错误恢复的生产工作流。

### 10.5 CrewAI 与 AutoGen

- **CrewAI**：以角色、任务和流程为中心，适合职责明确的多 Agent 团队。Sequential 简单可控；Hierarchical 由 Manager 分配任务，更灵活但更难预测。
- **AutoGen**：以 Agent 间对话为中心，支持 Two-Agent、Group Chat、Nested Chat 和代码执行代理，适合对话式协作和实验。

CrewAI 的 Sequential Process 依次执行明确 Task；Hierarchical Process 由 Manager Agent 分配和检查任务，灵活但行为更难预测。每个 Agent 应只绑定自身所需工具，Task 通过显式上下文依赖传递结果。

AutoGen 的 AssistantAgent 侧重模型推理，UserProxyAgent 可代表用户、执行代码或触发人工输入；Group Chat 需要 Speaker Selection 和终止条件，Nested Chat 可把一个复杂子对话封装为主对话中的一次调用。

两者都可能出现对话膨胀和循环，需要严格终止条件、消息裁剪、工具权限和成本预算。

### 10.6 其他常见选项

- **LlamaIndex**：数据接入、索引和 RAG 能力突出。
- **Spring AI/LangChain4j**：适合 Java/Spring 企业技术栈。
- **Dify**：低代码工作流和应用管理，适合快速产品化。
- **Semantic Kernel**：插件化地把模型能力接入企业应用。

### 10.7 选型原则

| 需求 | 优先选择 |
|---|---|
| 简单、性能敏感、强控制 | 直接模型 API |
| 快速 Chain/RAG 原型 | LangChain 或 LlamaIndex |
| 有状态、循环、审批、恢复 | LangGraph |
| Java/Spring 体系 | Spring AI / LangChain4j |
| 角色驱动多 Agent | CrewAI |
| 对话驱动多 Agent/代码执行实验 | AutoGen |
| 低代码应用搭建 | Dify |

选型前先做最小验证：同一批用例比较正确率、P95 延迟、成本、调试体验、升级风险和团队熟悉度。

框架版本变化快，应锁定依赖并记录实际版本。项目介绍时要能解释底层状态、循环和工具调用，不要只会背框架 API；否则一旦升级或更换框架，就无法定位问题。

## 11. Multi-Agent 协作

### 11.1 何时需要多 Agent

适合：任务可以自然分工、需要不同工具权限、多个专业视角能提升质量、子任务可并行。

不适合：任务简单、上下文高度共享、沟通成本超过分工收益，或单 Agent 加工具即可完成。

多 Agent 的收益来自专业化、并行和相互校验；它不会凭空增加事实。若多个 Agent 使用相同模型、相同 Prompt 和相同证据，它们的错误高度相关，简单投票不能显著提升可靠性。

### 11.2 常见架构

#### Supervisor

中心 Agent 负责拆解、分配、汇总和终止。优点是控制集中、易审计；缺点是 Supervisor 成为瓶颈和单点。

#### Peer-to-Peer

Agent 直接协商和转交任务。扩展灵活，但更容易产生循环、冲突和上下文膨胀。

#### Pipeline

需求 → 设计 → 实现 → 测试 → 审核。路径确定，最易工程化。

#### Debate

多个 Agent 独立提出观点，相互质疑，Judge 汇总。适合复杂推理和审查，但成本高，且多个 Agent 可能共享相同偏差。

### 11.3 任务分解与调度

任务对象至少包含：

```json
{
  "task_id": "T3",
  "goal": "生成最终报告",
  "dependencies": ["T1", "T2"],
  "assigned_agent": "writer",
  "status": "pending",
  "deadline": null,
  "input_refs": [],
  "output_ref": null,
  "retry_count": 0
}
```

调度器应支持依赖 DAG、并行执行、超时、取消、重试、优先级、资源限制和动态重新分配。

动态分配可依据 Agent 能力标签、历史成功率、当前负载、数据权限和预计成本。不能只让 Supervisor 根据角色名称自由选择，否则会出现热门 Agent 过载或越权分配。

### 11.4 通信与状态

消息应结构化，至少有 sender、receiver、task_id、message_type、content、artifact_refs、timestamp 和 trace_id。

共享全部对话简单但昂贵，也可能泄露私有信息。更好的方式是：每个 Agent 保留私有上下文，只向共享状态写入任务状态、最终结果、证据引用和必要决策。

通信消息应区分 `request`、`result`、`error`、`progress`、`handoff`、`approval_required` 等类型。所有异步消息携带 `trace_id`、`task_id` 和因果父消息 ID，才能还原跨 Agent 调用链。

### 11.5 结果聚合与冲突处理

- 多数投票：适合有明确候选答案的任务。
- 加权投票：依据 Agent 历史表现或领域可信度加权。
- 规则验证：代码、Schema、事实或约束可以确定性检查。
- Judge 模型：根据评分标准比较候选答案。
- 人工仲裁：高风险或置信度低时使用。

Judge 不能只看最终文风，应检查证据、约束满足度和可执行结果。

冲突处理可拆为：

1. **检测**：找出事实不一致、互斥结论和约束冲突。
2. **定位**：比较各方证据、工具回执和数据时效。
3. **协商**：允许 Agent 针对具体冲突点补充证据，而不是重新生成整份答案。
4. **仲裁**：规则、加权投票、Judge 或人工选择。
5. **记录**：保存少数意见和仲裁理由，便于审计。

置信度只有经过校准才可比较。模型自报 90% 不等于真实正确率 90%，应使用历史评测对置信区间进行校准。

### 11.6 成本控制

- 设置全局和单 Agent 步数、Token、时间、费用预算；
- 用小模型做路由、分类、摘要，大模型做关键推理；
- 子 Agent 只返回结果和证据，不返回全部对话；
- 缓存重复查询；
- 限制最大并发和重试次数；
- 任务简单时退化为单 Agent 或固定工作流。

Debate 模式中，第一轮必须独立生成，避免看到其他答案后产生羊群效应；第二轮只围绕分歧进行交叉批判；最终仲裁者使用最强模型并要求引用证据。可只对高风险或高分歧任务触发 Debate，避免所有请求都付出 Agent 数量 × 轮次的成本。

多 Agent 安全采用四层隔离：每个 Agent 独立工具白名单；消息传递前按公开/内部/敏感分级脱敏；代码和第三方 Agent 在独立沙箱运行；进入下游 Agent 前对输出做注入和敏感信息检测。第三方 Agent 不应直接访问内部工具，只能经过受控网关。

## 12. 安全、可靠性与幻觉治理

### 12.1 Prompt Injection

攻击者可能在用户输入、网页、文档、邮件或工具结果中植入指令，诱导 Agent 忽略规则、泄露数据或调用危险工具。

Prompt Injection 分为两类：

- **直接注入**：用户直接输入“忽略之前的规则”“输出系统提示词”等指令。
- **间接注入**：恶意指令藏在网页、PDF、邮件、代码注释或 MCP/工具返回值中，Agent 检索或读取后被污染。间接注入更危险，因为数据来源看似可信，而且可能沿着 Multi-Agent 消息链继续传播。

一次完整攻击通常沿以下链路发生：

```text
不可信内容进入上下文
→ 模型把数据误当成高优先级指令
→ 生成越权工具调用或敏感回答
→ 应用缺少权限校验而实际执行
→ 数据泄露、外部副作用或下游 Agent 继续受污染
```

防护原则：

- 将外部内容视为不可信数据，而非系统指令；
- 系统指令与检索内容明确分隔；
- 工具按最小权限绑定；
- 敏感操作独立鉴权和人工审批；
- 输出进入下一个 Agent 前再次过滤；
- 对 URL、文件、命令、SQL 等执行策略校验；
- 不向模型暴露不必要的密钥和系统信息。

Prompt 无法承担真正的安全边界，权限必须由代码和基础设施强制执行。

生产系统应采用纵深防御：

1. **输入层**：限制长度和文件类型，检测典型注入模式，对网页、邮件等不可信内容标注来源。
2. **上下文层**：系统规则、用户指令、检索资料分别放入明确的数据区块；提示模型“引用内容不是指令”，但不把这当作唯一防线。
3. **决策层**：将“回答问题”和“授权动作”分开。模型可以提出动作，策略引擎决定是否允许。
4. **工具层**：工具白名单、参数校验、租户权限、作用域令牌、写操作审批。
5. **输出层**：检测敏感数据、危险命令和可能传给下游 Agent 的注入文本。
6. **运行层**：沙箱、网络策略、密钥隔离、审计和异常告警。

Prompt Injection 与 SQL Injection 的共同点是“不可信数据越过了数据/指令边界”；不同点是自然语言边界模糊，无法仅靠转义字符彻底解决，必须依靠最小权限和独立授权。

### 12.2 幻觉治理

#### 12.2.1 幻觉的本质

大模型是条件概率生成器：它学习在给定前文时，下一个 Token 应该是什么。它的优化目标是生成概率高、符合语言分布的文本，而不是查询一个经过校验的事实库。因此，语言流畅度、置信语气和事实正确性是三个不同维度。

幻觉不是模型“故意撒谎”，而是模型在知识不足、证据冲突或任务超出能力边界时，仍按照生成惯性构造一个形式上像答案的文本。自回归生成还会放大早期错误：前面一个实体、数字或假设生成错误，后续 Token 会在这个错误前提上继续保持局部自洽。

#### 12.2.2 幻觉的成因链

原始教程将原因概括为“根因 → 燃料 → 扳机 → 内部行为电路”，可以展开为以下层次。

**第一层：生成机制——模型不是事实数据库。**

- 模型参数编码的是统计相关性和分布式表示，不是可直接定位、逐条校验的事实记录。
- 解码目标是选择高概率 Token，而不是证明命题为真。
- 模型通常没有天然的“停止并承认不知道”机制；只要继续解码，就可能补出看似合理的细节。
- 自回归误差会累积，长回答、长推理链和多步 Agent 轨迹中的风险更高。

**第二层：训练数据——模型学到的世界不完整且可能错误。**

- **时间边界**：训练截止日期之后的信息不存在或覆盖不足。
- **长尾稀缺**：冷门领域、少数语言、企业内部知识样本不足。
- **分布不均**：主流观点和高频表达被过度学习，少数但正确的事实可能被淹没。
- **数据污染**：互联网包含错误信息、互相抄袭、虚假引用和刻意投毒。
- **事实冲突**：不同时间、地区或来源的结论不一致，但上下文没有说明适用条件。
- **记忆式过拟合**：模型记住片段，却未必学会在变体问题上正确组合。

**第三层：训练与对齐偏好——模型倾向于“尽量回答”。**

- 预训练语料中“提问后给出回答”的模式远多于“拒绝回答”，形成继续生成的惯性。
- SFT、RLHF、DPO 通常奖励清晰、完整、乐于帮助的答案。如果数据没有充分奖励校准和拒答，就会强化不确定时仍给答案的倾向。
- 用户持续质疑或问题含错误前提时，模型可能为了顺从用户而改口，即 sycophancy（谄媚/过度顺从）。

**第四层：推理与解码——运行时设置会触发或放大错误。**

- temperature、top-p 较高会增加低概率 Token 被采样的机会。
- 问题含糊、缺少约束或要求异常具体时，模型可能自行补全缺失条件。
- 超长上下文会产生“中间信息遗忘”、注意力稀释和指令冲突。
- 复杂多跳推理中，中间步骤无法验证，局部错误会传递到最终答案。
- 模型自我反思并非独立事实源，可能用同一错误知识为原答案辩护。

**第五层：Agent 系统——外部组件也会制造“系统级幻觉”。**

- RAG 没召回正确文档、召回了过期文档或 Reranker 过滤掉了证据。
- 工具参数错误、API 返回异常、SQL 语义不符，但模型把结果当真。
- 摘要压缩丢失限定条件，长期记忆写入了错误事实。
- 多 Agent 共享同一错误前提，投票只会放大相关性错误，而不是形成独立验证。
- 引用生成与证据没有绑定，出现“答案大致正确但引用不存在”或“引用真实但不支持结论”。

有研究从神经元激活角度观察到少量与幻觉、过度顺从高度相关的单元，常被概括为 H-Neurons。其启发是可以研究内部风险监测，但这仍是前沿方向，不能据此声称存在一个可简单删除的“幻觉开关”；相关行为与正常语言生成能力可能纠缠。

#### 12.2.3 幻觉类型

| 类型 | 表现 | 主要定位方向 |
|---|---|---|
| 事实性幻觉 | 编造人物、日期、数字、事件 | 知识边界、检索和事实核查 |
| 引用幻觉 | 编造论文、链接、条款，或引用不支持结论 | 引用绑定与来源验证 |
| 推理幻觉 | 中间逻辑错误但结论语气自信 | 分步验证、计算工具、规则检查 |
| 指令幻觉 | 漏掉或误解用户约束 | 上下文管理、结构化约束检查 |
| 工具幻觉 | 调用不存在的工具、编造参数或结果 | Tool Schema、执行结果校验 |
| 记忆幻觉 | 把未发生的历史当成用户记忆 | 记忆来源、置信度和写入审核 |
| 行动幻觉 | 声称“已经发送/删除/支付”，实际未成功 | 以工具回执和事务状态为准 |

#### 12.2.4 分层治理方案

幻觉治理不能只靠一句“不要编造”，应把措施放在信息、推理、执行和验证各层。

1. **知识层**：用搜索、数据库、API 和 RAG 提供最新、私有、可追溯事实；对文档做版本和权限管理。
2. **检索层**：混合检索、Query 改写、Reranker；为“无答案”设置相关性阈值，不强行把弱相关片段交给模型。
3. **生成层**：要求答案只基于证据，区分“证据明确”“合理推断”“信息不足”；降低事实任务的随机性。
4. **结构层**：使用 Schema、枚举、类型、范围和业务规则校验，防止模型用自然语言掩盖格式错误。
5. **工具层**：数学交给计算器，代码交给测试，事实交给搜索，SQL 交给解析器和只读执行环境。以真实工具回执作为动作是否完成的唯一依据。
6. **验证层**：将关键断言拆成 claim，逐条检查是否有证据支持；校验 URL、文献标识、日期、金额和实体是否真实存在。
7. **冗余层**：对高价值任务生成多个独立候选或使用不同模型复核。必须保证信息源或推理路径具有独立性，否则多数票可能一致地错。
8. **交互层**：缺少关键参数时询问用户；信息不足时明确拒答；将事实与建议分开表达。
9. **人工层**：医疗、法律、金融、支付、删除和外部发布等高风险任务加入专业人员审批。

#### 12.2.5 一个可落地的事实校验链路

```text
生成草稿
→ 抽取原子断言（人物、时间、数字、因果、引用）
→ 为每条断言查找证据
→ 判定 supported / contradicted / insufficient
→ 删除或改写无证据断言
→ 重新生成带来源答案
→ 高风险内容进入人工审核
```

拒答也需要评测：阈值过低会增加幻觉，阈值过高会让系统对可回答问题频繁说不知道。应同时监控幻觉率、正确拒答率和错误拒答率。

幻觉不能被完全消除，系统目标应是“检测、约束、验证和安全降级”。

### 12.3 权限与副作用

将工具分级：

- L0：纯文本生成；
- L1：只读查询；
- L2：可撤销写操作；
- L3：外部通信、支付、删除、发布；
- L4：系统管理和高敏数据。

L2 以上应考虑二次确认、审批、幂等、审计和补偿。模型不能自行提升权限。

### 12.4 安全沙箱

沙箱应限制：

- 文件系统和宿主机访问；
- 网络出口；
- CPU、内存、进程数和执行时间；
- 系统调用与 Linux capabilities；
- 环境变量和凭据；
- 可安装依赖；
- 产物大小与类型。

对多租户代码执行，可使用每任务临时容器、MicroVM 或专用沙箱服务，并在任务结束后销毁环境。

### 12.5 可靠性模式

- 超时、重试和熔断；
- 幂等和去重；
- 缓存与降级；
- 状态 Checkpoint；
- Saga 补偿；
- Dead Letter Queue；
- 限流和背压；
- 人工接管；
- 混沌测试和故障演练。

## 13. 评测与可观测性

### 13.1 为什么 Agent 评测更难

传统函数通常输入确定、输出确定；Agent 路径和结果可能随模型采样、工具状态和外部数据变化。只看最终答案无法定位是模型、检索、工具、规划还是工作流出了问题。

因此需要分层评测：

1. **模型层**：理解、分类、生成、结构化输出。
2. **检索层**：召回和重排。
3. **工具层**：选择、参数、执行结果。
4. **轨迹层**：路径效率、无效步骤、循环。
5. **结果层**：正确性、完整性、业务成功率。
6. **系统层**：延迟、成本、稳定性、安全。

结果正确也不代表轨迹健康。例如 Agent 用错误工具碰巧得到正确答案，或重复调用十次后成功；这类系统一旦环境变化就容易失败。因此既要评价最终结果，也要评价是否选择了合理工具、参数是否正确、步骤是否必要、是否遵守审批与终止规则。

### 13.2 离线评测

建立版本化数据集，包含：

- 正常任务；
- 边界与缺参任务；
- 工具失败；
- 知识库无答案；
- 长上下文；
- 多轮纠错；
- Prompt Injection；
- 越权请求；
- 超时和高并发场景。

评价方法可组合：规则、单元测试、业务模拟器、人工标注和 LLM-as-a-Judge。LLM Judge 需要明确 rubric，并用人工样本校准，防止偏爱长答案或特定文风。

Golden Dataset 是持续演进的工程资产：

- 每个线上 Bug 修复后添加对应回归 Case；
- P0 核心用例要求全部通过，P1/P2 用例允许设定统计阈值；
- 保存输入、环境、知识版本、预期结果和评分规则；
- 定期清理过时事实并补充新业务分布；
- 训练/调优集与最终测试集分离，避免针对测试集过拟合。

对非确定性任务，应重复运行同一 Case，统计均值、方差和最差分位，而不是只跑一次。工具类任务可以使用模拟器或测试租户，避免评测时真的发送邮件、支付或删除数据。

#### 轨迹评测方法

- **精确匹配**：适合严格固定的状态机路径。
- **编辑距离/Jaccard**：比较工具序列和步骤集合。
- **语义相似度**：比较自然语言计划或决策摘要。
- **约束检查**：是否调用禁用工具、是否漏过审批、是否超过步数。
- **LLM Judge**：评价路径合理性，但应提供工具定义、目标和评分 rubric。

轨迹不必与唯一“标准路径”完全相同。只要满足约束并高效完成任务，替代路径也可视为正确。

### 13.3 核心指标

- Task Success Rate；
- 工具选择准确率；
- 参数正确率；
- 平均/最大步骤数；
- 无效调用率、重试率；
- 幻觉率、引用正确率；
- 人工接管率；
- P50/P95/P99 延迟；
- 每任务 Token 与费用；
- 安全攻击成功率；
- 用户满意度和业务转化指标。

指标之间存在张力：提高拒答率可能降低幻觉但损害任务完成率；增加 Reranker 和复核可提升质量但增加延迟与成本。上线门槛应使用多指标约束，例如“成功率不得下降、幻觉率低于阈值、P95 延迟和单任务成本不超过预算”。

### 13.4 Trace 与日志

每次任务应有统一 `trace_id`。Span 可覆盖：模型调用、检索、工具、Agent、节点、数据库和外部 API。

记录字段：

- 输入输出摘要或受控原文；
- 模型和 Prompt 版本；
- 工具名、参数摘要和错误码；
- Token、费用、延迟；
- 状态转移；
- 重试和降级；
- 用户、租户和权限上下文；
- 产物引用和最终状态。

日志要脱敏，避免把密钥、PII 和完整内部推理直接写入日志。

一次 Trace 的层级可设计为：

```text
Agent Run
├── route_intent
├── retrieve
│   ├── query_rewrite
│   ├── dense_search
│   └── rerank
├── model_decision
├── tool_call
└── final_generation
```

异步任务和多 Agent 仍应传播同一个根 `trace_id`，并用 Span Link 表达队列消息与后续 Worker 的因果关系。排障顺序通常是“指标发现异常时间窗 → Trace 找到失败任务 → Span 定位具体模型/工具/检索节点”。

### 13.5 线上监控与反馈闭环

典型链路：

```text
线上 Trace/用户反馈 → 失败聚类 → 新增评测样本
→ 修复 Prompt/检索/工具/流程 → 离线回归 → 灰度发布
```

告警应关注错误率、超时率、P95 延迟、费用突增、循环任务、工具异常、拒答率突变和安全策略命中。

线上抽样应兼顾成本与问题捕获：正常流量低比例采样，错误、超时、高费用、安全命中和用户差评任务 100% 保留。将失败 Trace 一键转入数据集，形成“线上问题 → 离线复现 → 回归修复”的闭环。

## 14. 部署、性能与成本控制

### 14.1 服务拆分

生产系统可拆为：

- API Gateway：认证、限流、租户识别；
- Agent Orchestrator：状态图和任务编排；
- Model Gateway：统一模型路由、重试和费用统计；
- Tool Service：受控业务工具；
- Retrieval Service：索引、检索和重排；
- Memory Service：会话和长期记忆；
- Sandbox Worker：隔离执行代码；
- Queue/Worker：处理长任务；
- Observability：日志、指标、Trace 和评测。

小项目不必一开始全部微服务化，但边界应清晰。

Model Gateway 应统一处理供应商协议、模型别名、超时重试、速率限制、内容策略、Token 统计和降级，避免每个 Agent 节点各自接模型。Tool Service 则统一做鉴权、幂等和审计，模型不可直接连接生产数据库或内部管理 API。

### 14.2 推理服务选型

- **vLLM**：高吞吐、PagedAttention、连续批处理，常用于主流开源 LLM 在线服务。
- **TGI**：Hugging Face 生态集成良好。
- **Triton**：适合统一托管多种模型和复杂推理流水线。
- **llama.cpp**：适合 CPU、边缘设备和量化小模型。

选型看模型支持、硬件、并发、首 Token 延迟、吞吐、量化、运维能力，而不是只看基准峰值。

需要同时衡量：

- TTFT（Time to First Token）：用户多久看到第一个 Token；
- TPOT（Time per Output Token）：后续生成速度；
- 吞吐：每秒处理请求或 Token 数；
- 并发下 P95/P99 延迟；
- GPU 显存、利用率和 KV Cache 命中/占用；
- 目标任务精度和结构化输出成功率。

### 14.3 Kubernetes 部署要点

- 镜像使用非 root 用户和固定依赖版本；
- 用 Secret 管理凭据，禁止写入镜像；
- 设置 requests/limits；
- GPU 节点使用标签、污点和专用调度；
- Readiness/Liveness/Startup Probe 分工明确；
- 长任务通过队列与 Worker 解耦；
- 流式响应考虑连接超时、反向代理缓冲和优雅终止；
- 多租户使用 Namespace、NetworkPolicy、ResourceQuota 和独立服务账号；
- 关键状态外置，Pod 保持无状态或可恢复。

流式 Agent 还需要处理客户端断开：取消不再需要的下游请求，或将确实需要继续的长任务转入后台；Pod 终止前停止接收新连接，并给进行中的流留出优雅关闭时间。

### 14.4 弹性与冷启动

扩缩容指标不能只看 CPU，还可看队列长度、并发生成数、Token 吞吐、KV Cache 使用率和 P95 延迟。

降低冷启动：保留最小副本、预拉镜像、模型权重本地缓存、预热模型、分层模型路由和请求排队。

GPU 服务扩容慢于普通 Web 服务。可组合“预测扩容 + 最小热副本 + 队列削峰”；小模型或托管 API 可作为冷启动期间的临时降级，但必须评测输出质量差异。

### 14.5 发布与回滚

- CI：单元测试、Schema 测试、Prompt/Agent 回归、安全扫描。
- CD：开发 → 预发 → 小流量金丝雀 → 全量。
- 每次发布记录模型、Prompt、代码、工具 Schema 和知识库版本。
- 质量、延迟或成本超过阈值自动停止或回滚。

蓝绿部署切换快但资源成本高；金丝雀发布更适合逐步验证模型行为变化。

Agent 发布版本不是单一代码版本，而是一个组合：应用代码、模型、系统 Prompt、Tool Schema、Skill、知识库、Embedding/Reranker 和安全策略。任何一项变化都可能影响轨迹，因此 Trace 必须记录完整版本集合。

### 14.6 成本优化

Agent 单任务成本可近似拆为：

\[
Cost = \sum ModelCalls + Retrieval + Rerank + Tools + Compute + Storage
\]

优化顺序：

1. 减少不必要步骤和上下文。
2. 路由到合适大小的模型。
3. 缓存 Prompt 前缀、检索和确定性工具结果。
4. 压缩工具返回值和历史消息。
5. 并行无依赖任务，降低总时延。
6. 为任务设置硬预算并支持降级。
7. 自托管模型时提高批处理与 GPU 利用率。

成本治理应分成两条线：

1. **速率/配额管理**：限制每用户、租户和工具的请求频率、并发、Token 速率，解决资源被瞬时耗尽的问题。
2. **花费管理**：采集每次模型、检索、Reranker、工具和计算资源用量，换算单任务与累计费用，解决预算问题。

可以实现独立 `Cost Manager`：

```text
采集 usage → 按模型/节点/租户计费 → 累加会话成本
→ 比较软预算与硬预算 → 预警/降级/停止 → 写入审计
```

- 软预算达到后可压缩上下文、减少候选、换小模型或停止 Debate。
- 硬预算达到后终止非关键步骤，并向用户说明当前产物和未完成部分。
- 若连续步骤没有带来新信息，即使未达到预算，也应根据边际收益提前停止。

模型路由可以使用“规则优先 + 小模型兜底”：明显的分类、格式化和摘要走小模型；复杂推理、冲突仲裁和关键生成走强模型。路由器本身也有延迟和费用，应监控误路由率以及降级后的质量损失。

## 15. 完整系统设计方法

系统设计题或真实项目可按以下顺序展开。

### 15.1 需求澄清

- 用户是谁，核心任务是什么？
- 数据源和工具有哪些？
- 实时性、准确性、并发和成本要求？
- 哪些动作会产生副作用？
- 是否需要人工审批？
- 是否多租户，权限如何隔离？
- 成功指标和不可接受风险是什么？

### 15.2 总体架构

```text
用户/客户端
    ↓
API Gateway（认证、限流、租户）
    ↓
Agent Orchestrator（状态、路由、计划、终止）
    ├── Model Gateway
    ├── Retrieval/RAG
    ├── Memory
    ├── Tool Services
    ├── Sandbox/Workers
    └── Human Approval
    ↓
Observability + Evaluation + Audit
```

### 15.3 企业客服 Agent 示例

关键链路：

1. 识别意图和风险等级。
2. FAQ/政策问题走 RAG，并返回引用。
3. 订单问题调用只读订单工具。
4. 退款等写操作检查身份和规则，并进入审批节点。
5. 工具失败时降级或转人工。
6. 保存会话状态和必要用户偏好。
7. 记录 Trace，使用真实会话持续构造评测集。

意图识别不能只返回一个标签，还应输出关键实体、缺失参数、风险等级和候选路由。对低置信度意图应澄清，而不是直接执行。知识答案必须使用当前生效版本的政策文档；业务状态以订单、物流和支付系统的实时回执为准，不能从知识库推断。

客服系统的记忆只保存确有长期价值的信息。当前订单号、验证码等短期敏感数据不应写入长期记忆；用户明确的语言、通知和服务偏好可以经确认后保存。

重点防护：跨用户订单越权、文档 Prompt Injection、虚假退款承诺、重复退款、敏感信息泄露。

### 15.4 Deep Research Agent 示例

```text
澄清研究问题 → 制定计划 → 生成搜索查询 → 多源搜索
→ 抽取证据 → 去重与可信度评分 → 识别信息缺口
→ 继续搜索或停止 → 生成带引用报告 → 事实复核
```

搜索结果分析节点至少完成三项工作：

1. **事实抽取**：把网页段落转成可独立验证的 Claim。
2. **信源评估**：记录作者/机构、发布时间、原始或二手来源、是否有利益冲突。
3. **矛盾检测**：同一 Claim 存在冲突时保留双方证据，继续查找更权威或更新来源。

可为来源建立启发式评分：官方原始材料和同行评审论文高于匿名转载；最新信息不一定更权威，时效权重应按问题类型设置。最终报告需要区分已证实事实、存在争议的观点和研究者推断。

防止搜索死循环：最大轮数、边际新信息阈值、重复 URL 检测、预算上限和用户中断。

保证客观性：来源多样性、时间和作者记录、区分事实与观点、展示冲突证据、每个关键结论绑定引用。

停止搜索不只看最大轮数，还可看边际信息增益：新一轮没有新增关键 Claim、来源高度重复、核心问题已有多源支持时停止；若预算将尽但仍存在信息缺口，应在报告中明确列出未解决问题。

### 15.5 可扩展性与故障处理

- API 和编排服务无状态扩容；
- 长任务进入队列；
- 状态、Checkpoint 和产物外置；
- 模型、搜索和工具提供方支持多供应商降级；
- 按租户限流和计费；
- 对关键写操作使用幂等和补偿；
- 对热点知识和 Prompt 使用缓存；
- 全链路 Trace 贯穿异步任务。

## 16. 项目实践路线

建议完成一个“企业知识与业务操作助手”，分阶段实现。

### 阶段 1：最小问答

## 18. 工程基础与前沿延伸

本章压缩原教程中的语言、数据库、中间件、分布式系统和 Coding Agent 内容，只保留与 Agent 工程直接相关的部分。

### 18.1 Python 异步编程

Agent 大量等待模型、搜索、数据库和外部 API，属于 I/O 密集型系统。Python `asyncio` 通过事件循环调度协程：协程执行到 `await` 时让出控制权，I/O 完成后由事件循环恢复执行。

```python
async def gather_context():
    docs, profile = await asyncio.gather(
        search_knowledge_base(),
        load_user_profile(),
    )
    return docs, profile
```

注意事项：

- 只有无依赖任务才能并行；
- 阻塞库会卡住事件循环，应换异步客户端或放入线程池；
- 使用超时、信号量和连接池限制并发；
- 请求取消时向下游传播取消信号；
- CPU 密集任务交给进程池或独立 Worker。

### 18.2 Java/Spring 技术栈

Java 团队可通过 Spring AI 或 LangChain4j 接入模型、向量库和工具。生产接入仍应遵循 Spring 的常规工程方法：

- Controller 只处理协议，Agent 编排放在 Service；
- Tool 与领域服务解耦，使用明确 DTO 和校验；
- WebFlux 适合流式输出和大量 I/O 并发，但不要在事件线程执行阻塞调用；
- 使用 Resilience4j 实现超时、重试、限流和熔断；
- 配置、密钥和模型路由统一管理；
- Micrometer/OpenTelemetry 采集指标与 Trace。

### 18.3 数据存储的职责划分

| 存储 | 在 Agent 系统中的典型用途 |
|---|---|
| MySQL/PostgreSQL | 用户、权限、任务、Checkpoint、审计、精确业务数据 |
| Redis | 会话缓存、限流、分布式锁、短期状态、幂等键 |
| 向量数据库 | 非结构化知识和长期语义记忆 |
| Elasticsearch/OpenSearch | 全文检索、过滤、混合检索和日志搜索 |
| MongoDB | Schema 变化频繁的文档或中间产物 |
| 对象存储 | 原始文件、长工具结果、报告和模型产物 |

数据库设计原则：精确状态以事务数据库为准，向量库只是派生索引；数据更新或删除后，应通过可靠事件同步重建或删除向量索引。

查询性能重点关注索引、执行计划、慢 SQL、连接池和大表治理。Agent 可能在短时间触发大量工具调用，因此必须限制查询并发、扫描行数和连接占用。

### 18.4 Redis 与分布式锁

Redis 常用于：

- 缓存模型和工具结果；
- 保存短期会话与任务进度；
- 令牌桶/滑动窗口限流；
- 幂等键和去重；
- 短期分布式锁。

分布式锁应使用唯一 owner token，并通过原子脚本“只允许持有者释放”；设置合理过期时间，长任务需续租。锁不能替代数据库唯一约束、事务和业务幂等。

缓存要处理穿透、击穿和雪崩：空值/布隆过滤器、互斥重建、随机过期、热点预热和多级缓存。知识库与 Prompt 缓存还必须把模型版本、租户、权限和数据版本纳入缓存键。

### 18.5 消息队列与最终一致性

耗时 Agent 任务应通过 Kafka、RocketMQ、RabbitMQ 等消息队列异步执行。基本链路：

```text
API 接收任务 → 数据库记录任务 → 发布消息 → Worker 执行
→ 保存产物/状态 → 推送或轮询结果
```

可靠性要点：

- 生产者确认和持久化；
- 消费者幂等；
- 重试队列与死信队列；
- 消息积压和消费延迟告警；
- 使用业务键保证同一任务的顺序；
- 大产物存对象存储，消息只传引用。

数据库状态与消息的一致性可使用事务消息、Outbox/本地消息表或 CDC。不要采用“先提交数据库，再直接发消息且不补偿”的脆弱双写。

### 18.6 分布式系统基础

Agent 系统需要接受以下事实：网络会超时、请求会重复、消息可能乱序、服务可能部分失败。

因此设计时应明确：

- **幂等**：同一动作执行多次与一次效果一致；
- **一致性边界**：哪些操作必须强一致，哪些可以最终一致；
- **超时预算**：总任务时间如何分配给模型、检索和工具；
- **重试边界**：只重试瞬时错误，避免重试风暴；
- **补偿**：多步骤写操作失败时如何撤销或修正；
- **可恢复状态**：实例宕机后从 Checkpoint 继续；
- **背压**：下游变慢时限制上游流量。

### 18.7 Harness Engineering

Prompt Engineering 关注单次指令，Harness Engineering 关注包围模型的完整运行环境。Harness 包括：

- 上下文选择与压缩；
- 工具和权限；
- 计划、循环和终止；
- 文件、终端、浏览器和沙箱；
- 子 Agent；
- 测试、反馈和恢复；
- 用户审批与审计。

同一模型在不同 Harness 中表现可能差异很大。优秀 Harness 会让任务状态外置、反馈快速可见、失败可恢复，并通过确定性验证减少模型自行判断的部分。

### 18.8 Agentic Coding

Coding Agent 不只是代码补全，而是围绕仓库执行“理解任务 → 搜索代码 → 修改 → 运行测试 → 根据错误继续修复”的闭环。

关键能力：

- 高效搜索文件、符号和引用；
- 读取仓库规则与构建命令；
- 生成小而可审查的补丁；
- 使用编译器、测试和静态分析获得确定性反馈；
- 在沙箱中执行命令；
- 保留任务进度、关键决策和未完成事项；
- 高风险命令和外部操作需要审批。

循环工程（Loop Engineering）的核心不是让模型无限重试，而是构造高质量闭环：每轮都有明确目标、可观察反馈、状态更新和停止条件。

### 18.9 自进化 Agent 的现实边界

所谓自进化通常不是模型自动修改自身参数，而是系统根据历史 Trace 和反馈更新：

- Prompt；
- Skill；
- 路由规则；
- 工具描述；
- 记忆和案例库；
- 评测数据；
- 少量微调适配器。

所有更新都应经过离线评测、权限审核、版本控制和灰度发布。允许 Agent 直接修改自己的核心规则、权限或生产代码，会形成难以审计的反馈回路。
